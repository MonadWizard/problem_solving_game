# The Abyss Expansion Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Surface the existing `roles`/`recency` fields in the Abyss (journey 3) UI, raise the hard-problem
share per company from 10 to 20 (16 easy / 24 medium / 20 hard = 60/company, 6000 total), add a third
"recently popular" recency bucket, and source a slice of hard problems from HackerRank alongside LeetCode.

**Architecture:** All curriculum content is generated by `scripts/gen-abyss-data.mjs` into
`public/data/journey3.json` (golden rule 1 — components never hardcode content). This plan extends that
script's curated pools and per-company selection logic, regenerates the data file, then updates the two
UI components (`ProblemRow.tsx`, `BossCard.tsx`) that render journey-3 problems to surface the new fields.

**Tech Stack:** TypeScript, React, Vite, Vitest, plain Node scripts (no new dependencies).

## Global Constraints

- All curriculum content lives in `public/data/*.json`, generated by `scripts/gen-abyss-data.mjs` — never
  hand-edit `journey3.json` directly (per `.claude/skills/curriculum/SKILL.md`).
- No copyrighted IP in any player-facing text (not touched by this plan — no new story/flavor copy).
- Every new problem slug/title/URL/difficulty must be one the implementer has verified is real — this plan's
  curated pools were verified via web search/fetch against the source platform (LeetCode, HackerRank) during
  planning; do not add further problems without the same verification.
- `journey3` problems only need slugs unique **within an island**, not journey-wide (real interview problems
  legitimately repeat across companies).
- After any `journey3.json` edit, regenerate `supabase/seed.sql` via `node scripts/gen-seed.mjs`.

## Important context discovered during planning

**`scripts/gen-abyss-data.mjs` has a pre-existing latent bug**: `dedupe()` builds pool entries as
`{ slug, title, pattern }`, dropping `difficulty` even though its first parameter (named `name`, e.g.
`'easy'`) is clearly meant to carry it. `toProblem()` then reads `p.difficulty`, which is `undefined` for any
freshly-picked (non-`kept`) pool problem — producing `difficulty: undefined, xp: NaN` in the output. This is
currently invisible because every company's `kept` data (re-read from the current `journey3.json` on each
run) already fully satisfies today's quotas (16/24/10), so the buggy code path never executes. Verified by
temporarily bumping `EASY_N` to 17 and rerunning: the one freshly-picked problem came out with
`"difficulty": undefined, "xp": null` in the JSON. **This plan's new hard/medium picks would hit this bug
immediately if it isn't fixed first** — Task 2 fixes it before any new pools are added.

company's 24 kept mediums, so `toRemove` and `swapIn` are always the same size and the total never drifts

will already be ≥3 (the swapped-in entries are `kept` next time and their `.recency` survives via Step 4's
fallback), so `swapTarget <= 0` and this block is a no-op — idempotent.
